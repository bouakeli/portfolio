Janvier – Mars 2025

Noodle

Application web de sondages pour planifier des événements en trouvant le créneau idéal selon les disponibilités des participants — clone de Doodle

PHP JavaScript AJAX API REST JWT Twig SQL Docker GitLab
Contexte

SAÉ de 2ème année de BUT Informatique, réalisée en binôme. Objectif : concevoir une application web complète avec un back-end PHP structuré (MVC, routage, Twig) et un front dynamique en JavaScript pur + AJAX, sans framework front-end.

Description
  • Création et gestion de sondages de disponibilités (titre, durée, participants minimum…)
  • Invitation de membres, gestion des droits (propriétaire, administrateur, simple membre)
  • Créneaux horaires proposés par les administrateurs, disponibilités saisies par les membres
  • Calcul automatique du meilleur créneau selon les contraintes
  • Accès via code secret partageable (lecture seule sans compte)
  • Interface dynamique : gestion des créneaux et membres sans rechargement de page (AJAX)
  • API REST avec authentification JWT, bibliothèque réactive maison
  • Virtualisation via Docker (multi-conteneurs)
Mon rôle

Développement PHP du back-end (contrôleurs, services, repositories), mise en place de l'API REST avec JWT, dynamisation JavaScript/AJAX de la page de gestion d'un sondage, et virtualisation Docker du projet.

Apprentissages critiques mobilisés
AC 3.1 — Comp. 3

Application PHP structurée (MVC, routage, vues Twig), contrôleurs distincts par domaine.

AC 3.3 — Comp. 3

Authentification JWT, sessions PHP, accès différenciés selon les rôles (propriétaire/admin/membre).

AC 4.1 — Comp. 4

Schéma base de données normalisé : utilisateurs, sondages, créneaux, disponibilités.

AC 4.2 — Comp. 4

Mots de passe hashés, authentification JWT pour l'API REST.

AC 4.3 — Comp. 4

Interface dynamique AJAX, bibliothèque réactive JavaScript maison (TD7).

AC 5.2 — Comp. 5

Analyse du cahier des charges Noodle, formalisation des règles métier complexes.

AC 5.4 — Comp. 5

Virtualisation multi-conteneurs Docker, organisation en livrables successifs.

Points techniques clés

Framework PHP maison développé en TD (routage, conteneur de services, Twig), API REST + JWT, AJAX sans dépendance externe, bibliothèque réactive JavaScript, Docker Compose.

Hexagoras

Application web de questions / réponses et votes autour de sujets politiques

PHP SQL SCRUM HTML/CSS JavaScript MVC GitLab Trello Discord Miro
Contexte

Réalisé dans le cadre du BUT Informatique en parallèle des cours, découpé en 4 sprints Agile SCRUM avec une équipe de 5 étudiants. Objectif : concevoir une application web complète en conditions proches d'un projet professionnel.

Description
  • Réseau social politique : création de questions, réponses et votes
  • Espace administrateur avec gestion des utilisateurs et des phases
  • Pages de listes de questions par phase (brouillon, en cours, terminées)
  • Fonctionnalités de recherche et de filtrage
  • Architecture MVC, mots de passe cryptés, gestion des sessions PHP
Mon rôle — Scrum Master

Participation active à l'ensemble des fonctionnalités. Rôle de Scrum Master : animation des daily meetings Discord, revues de sprint, adaptation du backlog Trello en fonction des priorités.

Apprentissages critiques mobilisés
AC 3.1 — Comp. 3

Sessions PHP pour gestion utilisateurs (admin/utilisateur), pages dynamiques avec listes de questions par phase, navigation entre espaces.

AC 3.3 — Comp. 3

Accès pages restreints par sessions, mots de passe cryptés en base, rôles différenciés.

AC 4.1 — Comp. 4

BDD structurée (utilisateurs/questions/réponses/votes), jointures multiples pour affichage complet.

AC 4.2 — Comp. 4

Mot de passe crypté en BDD, sessions pour persistance de l'identité utilisateur.

AC 4.3 — Comp. 4

Listes questions filtrées par phase, pages de résultats votes, recherche et filtrage.

AC 4.4 — Comp. 4

Gestion de données hétérogènes : utilisateurs, questions, réponses, votes.

AC 5.2 — Comp. 5

Formalisation des besoins : user stories, backlog produit.

AC 5.4 — Comp. 5

4 sprints SCRUM planifiés, suivi GitLab issues, revues sprint.

Pokémon TCG

Reproduction du fonctionnement d'un jeu de cartes Pokémon en application Java

Java JavaFX POO Binôme GitLab
Contexte

Réalisé en binôme sur plusieurs mois, organisé en livrables progressifs : d'abord le moteur de jeu, puis l'interface graphique JavaFX.

Description
  • Reproduction des mécaniques TCG Pokémon (PV, attaques, K.O.)
  • Gestion du Pokémon actif, des dégâts, du K.O. et de la pioche
  • Interface graphique complète avec JavaFX
Mon rôle

Développement Java de la logique du jeu : classes Pokémon, gestion des PV, attaques, états vivant/K.O.

Participation à la création de l'IHM JavaFX pour relier les actions du joueur au code Java.

Apprentissages critiques mobilisés
AC 3.1 — Comp. 3

Logique de jeu Java (Pokémon, actifs, attaques), interface JavaFX connectée au moteur.

AC 4.3 — Comp. 4

Interface JavaFX : affichage des PV, état du Pokémon, pioche et banc.

AC 5.2 — Comp. 5

Cahier des charges binôme : mécaniques TCG formalisées.

AC 5.4 — Comp. 5

Livrables progressifs avec objectifs techniques intermédiaires.

Organisation en binôme

Répartition des tâches (logique de jeu / interface), échanges réguliers pour synchroniser le code. Avancement structuré en livrables successifs.
